# Как устроена группировка

## Вступление

Нагрузку на базы данных традиционно делят на OLTP и OLAP - транзакционную и аналитическую.
И если OLTP ассоциируют с UPDATE, а OLAP - это группировка и агрегация, о которой мы сейчас и поговорим.

## Как устроены агрегатные функции

Вначале вспомним, что такое агрегатная функция.

Агрегатная функция - это функция, вычисляющая результат на множестве данных.

> Слайд: эти 2 запроса

Например,

```sql
-- Самый большой возрасти среди всех пользователей
SELECT max(age) FROM users;

-- Среднее количество денежных переводов каждого пользователя
SELECT userid, avg(transferred) FROM user_transactions GROUP BY userid;
```

Агрегирующие функции бывают разные и у всех разное поведение.

Например, `max` мы можем рассчитывать налету, храня только 1 число (самое большое значение), а вот для эффективного расчета `avg` придется хранить 2 числа - сумма и количество, но при этом возвращаемое значение это 1 число, которое вычисляется из этих 2-ух.

> Слайд: псевдокод ниже

Чтобы поддерживать такие различные свойства был разработан свой протокол работы с агрегатами.
Визуализировать его можно так:

```python
# Инициализация состояния
state = init()

# Применение функции перехода
for tuple in input:
   state = transit(state, tuple)

# Подсчет результата
result = finalize(state)
```

1. Инициализируем состояние
2. Для каждого кортежа из входа применяем функцию перехода
3. Вызываем финализатор, подсчитывающий результат

> Слайд: create aggregate ниже

Есть много встроенных агрегатных функций, но можно создать и свои. Сделать это можно с помощью `CREATE AGGREGATE`. У нее много аргументов, но мы сконцентрируемся на главных:

```sql
CREATE AGGREGATE some_agg(a int, b int) (
   -- Начальное состояние
   initcond  = '(0,0)'
   -- Функция перехода
   sfunc     = average_transition,
   -- Финализатор
   finalfunc = average_final,
)
```

- `initcond` - изначальное значение агрегата
- `sfunc` - функция перехода
- `finalfunc` - финализатор

> Слайд: запрос и его результат на слайде

Для встроенных агрегатов мы можем посмотреть из чего они сделаны.
Все агрегаты хранятся в таблице pg_aggregate и вот небольшая часть его содержимого.

```sql
select aggfnoid, agginitval, aggtransfn, aggfinalfn from pg_aggregate;

         aggfnoid       |    agginitval   |    aggtransfn    |      aggfinalfn              
------------------------+-----------------+------------------+-----------------------
 pg_catalog.stddev      | {0,0,0}         | float4_accum     | float8_stddev_samp
 pg_catalog.avg         | {0,0,0}         | float4_accum     | float8_avg
 pg_catalog.avg         | {0,0}           | int4_avg_accum   | int8_avg
 pg_catalog.max         |                 | int4larger       | -
 pg_catalog.max         |                 | float4larger     | -
 
```

Основные моменты мы можем увидеть:

- `avg` для `int4` в initcond - 2 нуля (сумма и количество), функция перехода `int4_avg_accum` - увеличивает количество и сумму, а `int8_avg` финализатор - делит их.
- `max` для `double` не имеет финализатора и его состояние описывается одним числом

С помощью этих 3 вещей мы можем описать практически любую агрегирующую функцию.

Но агрегирующие функции лишь часть общей картины под названием группировка.

## Простая агрегация

> Слайд: запрос и вывод ниже

Для начала рассмотрим самый простой случай - плоская группировка. В оригинале его называют plain aggregate.

Сама плоская группировка - это группировка по всем данным, т.е. мы не указываем `GROUP BY` и в узле она отображается просто как `Aggregate`.

```sql
EXPLAIN SELECT avg(a) FROM tbl;
                         QUERY PLAN                          
-------------------------------------------------------------
 Aggregate  (cost=35.50..35.51 rows=1 width=8)
   ->  Seq Scan on tbl  (cost=0.00..30.40 rows=2040 width=4)
(2 rows)
```

Нам не нужно предобрабатывать входные данные. Просто получаем на вход кортеж и отдаем функции перехода. Разберем как это работает.
 
> Слайд: `ExecAgg` + `AggStrategy`

За выполнение узла группировки отвечает функция `ExecAgg`, внутри которой простой switch определяющий нужную стратегию.

Вы уже знаете, что PostgreSQL умеет группировать несколькими стратегиями. Каждая из них описана перечислением `AggStrategy`.

За плоскую группировку отвечает `AGG_PLAIN` и для выполнения используем `agg_retrieve_direct`. Остальные мы рассмотрим потом.

> Слайд: упрощенный код `agg_retrieve_direct` выделен fetch_input_tuple

PostgreSQL работает по итераторной модели и каждый раз возвращает по 1 кортежу из подузла. Если вернул `NULL`, то значит данных больше нет. И внутри нас ждет обычный `while` цикл проходящий по всем кортежам из входа и передающий их функции перехода.

Для чтения кортежей мы используем `fetch_input_tuple`. Пока нам не важно что внутри, но поверьте на слово, что они читаются из под-узла, например, `SeqScan`.

> Слайд: код `agg_retrieve_direct` выделен `initialize_aggregates`

После получения кортежа нам нужно инициализировать новую группу.

> Слайд: `initialize_aggregates` цикл простой

Выполняется это с помощью функции `initialize_aggregates`. Чтобы понять, что происходит разберем основные компоненты кода.

> Слайд: схема с инициализацией, где AggStatePerGroup - `init-scheme`

`AggStatePerGroup` хранит само состояние агрегата. Эта структура создается для каждого набора конкретных значений атрибутов группировки.

Как мы уже видели, для AVG состояние - это пара из 2 чисел - сумма и количество.

Но можете заметить, что здесь не 2 поля: значение и NULL, но есть и 3 - флаг `noTransValue`.
Дело в правилах работы с NULL'ами - если изначальное состояние агрегата NULL, то первое не NULL значение становится состоянием, но если мы используем только 1 флаг - как мы различим NULL, который еще не выставлен от NULL, который вернула функция перехода. Для решения мы задействуем дополнительный флаг.

> Слайд: схема с PerTrans (`trans-final-scheme`)

Для работы с агрегатами используется 2 структуры.

Одна из них `AggStatePerTrans`. Основные поля в ней - изначальное состояние и функция перехода.

Но вот финализация выделена в отдельную структуру - `AggStatePerAgg`. И опять вопрос - зачем. А все дело в оптимизации.

> Слайд: запрос

```sql
select agginitval, aggtransfn, count(*) cnt from pg_aggregate group by 1, 2 having count(*) > 1 order by 3 desc;
  agginitval   |          aggtransfn          | cnt 
---------------+------------------------------+-----
 {0,0,0,0,0,0} | float8_regr_accum            |  11
 {0,0,0}       | float4_accum                 |   7
               | ordered_set_transition       |   7
 {0,0,0}       | float8_accum                 |   7
               | int4_accum                   |   6
               | numeric_accum                |   6
               | int2_accum                   |   6
               | int8_accum                   |   6
               | ordered_set_transition_multi |   4
               | interval_avg_accum           |   2
               | numeric_avg_accum            |   2
               | int8_avg_accum               |   2
               | booland_statefunc            |   2
(13 rows)
```

Существует множество функций агрегации с одинаковыми начальным значением и функцией перехода. Это значит, что мы можем просто использовать разные финализаторы над одним и тем же состоянием и не тратить впустую ресурсы.

> Слайд: запрос и под ним скриншот отладчика `numaggs = 2, numtrans = 1`

И в качестве примера можно привести 2 функции: `avg` и `stddev`. Если запустим такой запрос, то окажется, что агрегатных функции 2 (`numaggs`), но состояние хранится только 1 (`numtrans`).

```sql
select avg(a::float), stddev(a::float) from tbl;
```

> Слайд: `initialize_aggregate`, код в конце

Теперь логика инициализации ясна: из `AggStatePerTrans` копируем состояние в `AggStatePerGroup` - само значение и 2 флага.

> Слайд: `agg_retrieve_direct` около `advance_aggregates`

Теперь нам необходимо эту самую функцию перехода применить. Делается это в функции `advance_aggregates`, но функция перехода не будет напрямую в ней вызываться.

Дело в том, что многие выражения в PostgreSQL выполняются не как отдельные функции, а преобразовываются в последовательность команд для выполнения. В терминах кода - выражения компилируются.

> Слайд: `ExecInterpExpr`, но только те шаги, которые нужны мне

Сейчас для того, чтобы вызвать функцию перехода нам нужно только получить значение из кортежа и вызвать саму функцию перехода.

> Слайд: скриншот отладчика с вызовом этой функции перед применением состояния (transdata->count++ бп)

Проваливаемся внутрь и видим непосредственный вызов функции перехода - `int4_avg_accum`. Для хранения состояния используется массив из 2 элементов - количество и сумма. Вначале они все инициализированы в 0.

> Слайд: скриншот отладичика в PG_RETURN_

После применения кортежа наше состояние изменилось. На вход на подали число 1, поэтому count равен 1 и sum тоже равен 1.

> Слайд: `agg_retrieve_direct` выделен fetch_input_tuple + break

Теперь нам осталось только читать кортежи и применять функцию перехода пока не закончится вход.

> Слайд: `agg_retrieve_direct` выделен finalize_aggregates

Когда вход закончился, то применяем финализатор.

> Слайд: `finalize_aggregate`, но только применение финализатора + скриншот отладчика с вызовом

Он вызывается в функции `finalize_aggregate`. Для `avg` - это `int8_avg`.

> Слайд: скриншот отладчика

В состоянии можем заметить, что подсчитаны все `count` и `sum`. Дальше остается их поделить, но для этого используются функции над типом numeric для сохранения точности и эту часть мы опустим.

## Группировка по столбцам

> Слайд: запрос с GROUP BY ниже

```sql
SELECT a, avg(b) FROM tbl GROUP BY a;

                       QUERY PLAN                        
---------------------------------------------------------
 HashAggregate
   Group Key: a
   ->  Seq Scan on tbl

SELECT a, b, c FROM tbl GROUP BY a, b, c;

                           QUERY PLAN                           
----------------------------------------------------------------
 Group  (cost=1.08..1.12 rows=3 width=12)
   Group Key: a, b, c
   ->  Sort  (cost=1.08..1.09 rows=4 width=12)
         Sort Key: a, b, c
         ->  Seq Scan on tbl  (cost=0.00..1.04 rows=4 width=12)
```

К этому моменту мы разобрали то, как работают функции агрегации и плоская группировка, но чаще выполняется группировка по атрибутам.

Для этого в SQL используется конструкция `GROUP BY`, в которой передается список из элементов группировки. Postgres поддерживает 2 стратегии: хэширование и сортировка.

### Сортировка

> Слайд: запрос с explain и GroupAggregate

```sql
EXPLAIN SELECT a, b FROM tbl GROUP BY a, b;

                  QUERY PLAN                           
----------------------------------------------
 Group
   Group Key: a, b
   ->  Sort
         Sort Key: a, b
         ->  Seq Scan on tbl
```

Вначале рассмотрим сортировку. В плане вы ее видели как GroupAggregate или просто Group.

> Слайд: кортежи отсортированы и показываем границы каждой группы (`group-sort-idea`)

Ее идея в следующем - если вход отсортирован, то элементы каждой группы находятся друг за другом, а первый неравный элемент эти группы разделяет.

Если текущий кортеж равен предыдущему - вызываем функцию перехода, иначе финализатор. Причем хранить нам нужно только представителя группы. Чтобы понять идею, вручную обработаем вот такую последовательность.

> Слайд: группа шаг 1

В самом начале у нас нет представителя, поэтому первый кортеж им и становится. `1 1 1`

> Слайд: группа шаг 2

Следующий кортеж равен предыдущему - мы в границе этой же группы, поэтому применяем функцию перехода.

> Слайд: шаг 3

Но вот сейчас на вход подается неравный кортеж - отличается 3-ий атрибут.

Поэтому мы финализируем группу `111` и начинаем новую, обновляя представителя.

> Слайд: шаг 4

Но очередной кортеж отличается, поэтому эта группа заканчивается практически сразу и начинается очередная.

> Слайд: шаг 5

Следующий кортеж равен предыдущему, значит мы снова внутри той же группы

> Слайд: шаг 6

И последний кортеж неравен представителю, поэтому опять заканчиваем текущую группу и начинаем новую.

> Слайд: шаг 7

Вход закончился и это значит, что мы должны финализировать оставшееся состояние.

Таким образом и устроена обработка с помощью сортировки. Мы не скакали из конца в начало и все что нам нужно хранить - это только представитель группы.

> Слайд: `ExecAgg`

`AGG_SORTED` представляет эту стратегию и как можете заметить, плоская группировка выполняет тот же самый код. И если так подумать, то плоская группировка это вырожденный случай сортировки, когда все кортежи как бы равны.

> Слайд: `agg_retrieve_direct`

Что происходит внутри `agg_retrieve_direct` мы только что видели, поэтому сконцентрируемся на основном различии - обнаружении границ групп. А заключается она вот в этой строчке:

> Слайд: `if (node->aggstrategy != AGG_PLAIN)`

Когда мы прочитали очередной кортеж, то нам нужно проверить, что мы не достигли конца группы. Если так, то просто прерываем цикл и вызываем финализатор.

Чтобы определить конец группы нам просто нужно выполнить обычную функцию равенства. Чтобы найти ее мы используем системный каталог.

> Слайд: `lookup_type_cache`

Но у типов могут быть разные свойства, например, какие-то можно сравнивать, можно хэшировать и т.д. Все это описывается классами и семействами операторов, но в нее мы углубляться не будем. Сделаем только вывод - тип может быть сравниваемым и/или хэшируемым и мы должны проверить все доступные для нас места - в частности функции равенства для BTree и HASH индексов.

Вы могли заметить, что функции сравнения из системного каталога помечены STRICT, то есть должны на все NULL возвращать false, но если мы запустим запрос с NULL атрибутами, то они все попадут в одну группу.

> Слайд: `EEOP_NOT_DISTINCT` как выполняется

На самом деле для сравнения используется конструкция `IS NOT DISTINCT FROM`, которая имеет правила сравнения с NULL и возвращает `true` если оба операнда `NULL` и false если только 1 из них. И обычный компаратор вызывается если оба операнда нормальные.

Таким образом мы находим границы групп.

### Хэширование

> Слайд: запрос explain с HashAggregate

```sql
EXPLAIN SELECT a, b FROM tbl GROUP BY a, b;

      QUERY PLAN      
------------------------
 HashAggregate
   Group Key: a, b
   ->  Seq Scan on tbl
```

Сортировка закончилась и мы переходим к хэшированию.

Оно описывается просто. Строим хэш-таблицу: ключ - атрибуты группировки, а значение - состояние агрегата. Читаем очередной кортеж, рассчитываем его хэш, получаем состояние из хэш-таблицы и обновляем агрегат. Но дьявол кроется в деталях.

> Слайд: тип - функция хэширования, напротив базового - builtin() (или типа того)

Первое, нам нужно понять как хэшировать атрибуты.

В PostgreSQL можно выделить 4 типа типа: базовый, диапазон, перечисление и составной.

Чтобы захэшировать диапазон - хэшируем начало и конец (`hash_range`), перечисление - само числовое представление (`hashenum`), а для составного типа (`hash_record`) - каждый из его членов.
Поэтому если мы умеем хэшировать базовый, то и все остальные.

> Слайд: запрос и результат

```sql
select ap.amproc
   from pg_opfamily f
      join pg_am a on f.opfmethod = a.oid
      join pg_amproc ap on ap.amprocfamily = f.oid
where a.amname = 'hash'
  and ap.amproclefttype = 'int4'::regtype
  and ap.amprocnum = 1;

  amproc  
----------
 hashint4
(1 row)
```

Чтобы получить функцию хэширования мы используем системный каталог. В частности, нам нужно посмотреть support функцию для HASH индекса, а именно первую функция поддержки. Для int это `hashint4`.

Откуда взять функцию хэширования разобрались и осталась главная проблема - нехватка памяти.

Для каждого узла задается его мягкий предел использования памяти - `work_mem`. И его превышение необходимо обработать. Для решения этой задачи используется знание о механизме работы хэш-таблицы.

> Слайд: хэш-таблица, от нее отходят стрелки с партициями, а в ней PerGroup с хэш-значениями

Верхнеуровнево, хэш-таблицу можно представить как массив, элементами которого являются партиции. В каждой партиции лежат элементы, хэши которых по определенной маске равны. Но так как эта часть хэша равна, то кортежи с одними и те ми ключами будут принадлежить одной и той же партиции.

Поэтому идея следующая - когда мы начинаем сброс, то разбиваем все множество на несколько отдельных непересекающихся партиций с помощью их хэша. Рассчитываем это количество таким образом, чтобы из каждой можно было построить новую хэш-таблицу полностью помещающуюся в памяти. В лучшем случае, эта стратегия позволит нам обойтись только 1 сбросом на диск.

> Код: функция `hash_choose_num_partitions`

Собственно эта логика и используется внутри функции расчета количества партиций.

На последней строке (`mem_wanted / spill_mem_limit`) мы и видим основную идею - рассчитывать размер партиции таким образом, чтобы каждая полностью умещалась в памяти - общий размер результирующей таблицы делим на предел памяти.

К сожалению мы можем ошибиться и придется выполнить еще один сброс. Так как мы уже использовали какое-то количество бит хэша и при этом все кортежи попали в одну и ту же партицию, то префиксы хэшей этих кортежей равны, поэтому их нет смысла рассматривать.

> Код: функция `hashagg_spill_init`, где расчет маски

Чтобы уметь обрабатывать такие ситуации используется не простой префикс, а обобщенная битовая маска. Они рассчитываются на основании того сколько нам требуется партиций и сколько битов мы уже использовали.

Но для того, чтобы корректно это рассчитать, нам нужно знать сколько будет групп. Изначальное значение нам дает планировщик, но дальше это использовать нельзя так как это совсем другая оценка.

> Слайд: HyperLogLog таблица и код, где добавляем его

Решение простое - мы будем сами это подсчитывать. Для этого используется структура данных HyperLogLog. При каждом сбросе на диск мы будем ее обновлять и таким образом к моменту расчета количества партиций у нас на руках будет оценка кардинальности.

Теперь на все вопросы мы ответили и можем рассмотреть как это происходит на самом деле. В начале рассмотрим только логику в памяти.

> Код: `ExecAgg`

Как и всегда, мы начинаем с функции `ExecAgg`, но на этот раз наша стратегия хэширование и за нее отвечает `AGG_HASHED`. Первое, что нужно сделать - заполнить хэш-таблицу.

> Код: `agg_fill_hash_table`

Этот код нам уже знаком: `fetch_input_tuple` - читаем кортеж и в `advance_aggregates` применяем функции перехода. Что они делают мы знаем. Единственное, что отличается - промежуточный этап `lookup_hash_entries`. В ней и находится логика работы с хэш-таблицей.

> Код: `lookup_hash_entries`

Сразу встречаем структуру `AggStatePerHash` - это контекст хэш-таблицы.

> Слайд: представление хэш-таблицы + AggStatePerHash структура и от нее отходят стрелки к функции хэширования, таблице и т.д.

Внутри находится сама хэш-таблица, а также вспомогательные данные, такие как функции хэширования и проверки равенства.
Устройство самой хэш-таблицы простое: ключ - атрибуты группировки, а значение - структура `AggStatePerGroup`, которую мы уже видели.

> Слайд: `entry = LookupTupleHashEntry`

Основная логика взаимодействия с хэш-таблицей заключена в `LookupTupleHashEntry`. Для оптимизации поиск и создание нового элемента в хэш-таблице выполняется за 1 вызов функции.

> Слайд: `LookupTupleHashEntry`

Внутри ничего необычного: рассчитываем хэш-значение кортежа, а затем вызываем саму функцию поиска с данным хэшем.

> Слайд: `TupleHashTableHash_internal`

Сначала рассчитываем хэш.

Откуда берутся функции хэширования мы уже обсудили. И для вызова самой функции используются компилируемые выражения.

Но что здесь приметно - это финальный вызов `murmurhash` для самого хэш-значения. Это требуется для получения более равномерного распределения хэш-значения, даже если пользовательская функция написана плохо.

> Слайд: `LookupTupleHashEntry_internal`

Теперь хэш на руках и мы идем искать элемент. Внутри опять ничего необычного - вызываем функцию поиска для самой структуры хэш-таблицы и, инициализируем запись, если нужно.

> Слайд: `lookup_hash_entries` выделен `initialize_hash_entry`

Все операции с хэш-таблицей завершены и мы возвращаемся обратно. Последнее, что надо сделать - инициализировать состояние агрегата.

> Слайд: `initialize_hash_entry`

Тут пока просто вызывается знакомая функция `initialize_aggregate` для каждой функции перехода.

> Слайд: `lookup_hash_entries` выделен `pergroup = ...`

В конце, мы в отдельное место сохраняем указатель на состояние, откуда его затем использует `advance_aggregates`. С помощью этого отдельного массива `pergroup` мы можем поддерживать самые разные стратегии и не учитывать то, как это состояние хранится.

> Слайд: `agg_fill_hash_table`

Группу нашли, агрегат инициализирован и мы вызываем `advance_aggregates` для применения функции перехода. Делаем так пока не закончится вход.

> Слайд: `ExecAgg`

Теперь таблица заполнена и мы можем читать из нее кортежи.

> Слайд: `agg_retrieve_hash_table_in_memory`

Внутри же обычная итерация по хэш-таблице и вызов финализатора.

### Хэширование, сброс на диск

Но работа полностью в памяти это самая легкая часть. Теперь нам нужно подумать о нехватке места, то есть как мы поступаем при сбросе на диск.

Первый вопрос - где и когда мы обнаружим переполнение? Самое оптимальное - проверять тогда, когда мы память выделили, то есть после создания новой группы.

> Слайд: `initialize_hash_entry`

Эту функцию мы уже видели - она вызывается каждый раз, когда создается новая группа и это тот самый момент, когда мы можем выполнить такую проверку.

Перед тем как инициализировать агрегаты вызывается `hash_agg_check_limits`, которая и выполняет эту проверку.

> Слайд: `hash_agg_check_limits`

Проверка довольно тривиальная - сравниваем сколько памяти выделили с пределом. И когда эта проверка проходит, то мы переходим в режим сброса - `spill mode`.

> Слайд: `hash_agg_enter_spill_mode`

Этот `spill_mode` можно назвать другим режимом выполнения, так как меняется не только то, что мы сбрасываем кортежи на диск, но вообще предположения о состоянии.

Во-первых, если нет памяти, то нужно сообщить об этом, чтобы функция перехода не вызывалась.

Одно из преимуществ, того, что мы компилируем выражения еще и в том, что мы, заранее зная контекст, можем изменить логику функции.

> Слайд: `lookup_hash_entries` часть с `pergroup[setno] = NULL`

Теперь, т.к. какой-то группы нет, то нет и состояния, соответственно, мы передаем в `pergroup` `NULL`. Но если у нас все помещается в памяти, то мы можем гарантировать, что никаких `NULL` там нет.

> Слайд: `hash_agg_enter_spill_mode` выделен `agg_recomile_expressions`

Поэтому, у каждого выражения функции перехода есть 2 версии: с проверкой на `NULL` и без. В начале мы вызываем код без проверки для большей производительности, но как только мы входим в spill режим, то создаем новую версию с этой проверкой.

> Слайд: `ExecBuildAggTransCall`

Если спустимся по колстеку, то дойдем до `ExecBuildAggTransCall` и первой командой, которую она добавляет и является эта самая проверка на `NULL`.

> Слайд: `hash_agg_enter_spill_mode` выделен `LogicalTapeSetCreate`

Далее, нам нужно подготовить место для сброса кортежей. Для этой цели используется такая абстракция как `LogicalTapeSet` - множество `LogicalTape`, логических файлов, в которые мы будем записывать кортежи. Создаем по одному LogicalTape на партицию.

> Слайд: архитектура LogicalTape и как они переиспользуются

Сам `LogicalTapeSet` - это всего-лишь удобная обертка над одним временным файлом, которая позволяет разделять его на несколько независимых виртульных файлов. Но интересна идея за ним - многие паттерны работы очень похожи:

1. Последовательно записываем на диск что-то
2. Затем полностью это считываем, без необходимости повторного чтения

Этот паттерн работы также используется и при работе с партициями, поэтому для этих случаев сделана небольшая оптимизация - после того, как мы прочитали какой-то блок данных из тейпа, то этот блок возвращается в пул свободных и другие могут его затем переиспользовать.

На слайде можете видеть пример - изначально в нашем файле было 2 партиции. Но затем мы начали читать из первой. После прочтения первой страницы она была освобождена. Но в процессе выполнения мы поняли, что ошиблись с оценкой и пришлось выполнить сброс на диск еще раз - это 3-я партиция. И вместо увеличения файла для записи новой страницы в конец мы переиспользуем освободившуюся страницу.

> Слайд: `hashagg_enter_spill_mode` выделен `hashagg_spill_init`

И последняя часть - инициализация самих партиций. Чтобы централизованно всё отслеживать используется отдельная структура `HashAggSpill`. В ней хранится вся информация по каждой партции.

Так как это наш первый сброс, то при инициализации статистику мы берем от планировщика - `numGroups` количество групп, а `0` - это количество использованных битов хэша (пока не использовали).

Внутри мы оцениваем количество партиций, которое нам потребуется и для каждой создаем свою партицию, в которую будем сбрасывать кортежи.

Подытоживая, spill mode означает следующее:

1. Перекомпиляций функций перехода с проверкой `NULL`
2. Создание временного файла для сброса кортежей
3. Подсчет статистики для каждой партиции

> Слайд: `lookup_hash_entries` выделен `p_isnew`

Представим, что мы вошли в этот режим и очередной кортеж отсутствует в хэш-таблице, то есть его придется сбросить.

Но если мы в режиме сброса, то новые элементы создавать не надо, и нам нужно об этом сообщить. Делается это с помощью той же самой переменной, которая нам говорит о том, что новый элемент был создан. Когда мы передаем туда NULL, то хэш-таблица понимает, что новые элементы создавать не нужно и проводит обычный поиск.

> Слайд: `lookup_hash_entries` выделен `spill_tuple`

Теперь же при поиске элемента нам могут вернуть `NULL` и тогда нам нужно сбросить этот кортеж. Сама логика сброса кортежа тоже довольно интересна.

> Слайд: `SELECT a FROM tbl GROUP BY a` где SeqScan много атрибутов возвращает

```sql
EXPLAIN (VERBOSE) SELECT a FROM tbl GROUP BY a;
                           QUERY PLAN                           
----------------------------------------------------------------
 HashAggregate  (cost=1.05..1.08 rows=3 width=4)
   Output: a
   Group Key: tbl.a
   ->  Seq Scan on public.tbl  (cost=0.00..1.04 rows=4 width=4)
         Output: a, b, c
(5 rows)
```

Во-первых, если вы видели `EXPLAIN VERBOSE` вывод, то знаете, что всякие `SeqScan` узлы отдают все атрибуты, а не только те, которые нужны нам и вышестоящим узлам.

> Слайд: `hashagg_spill_tuple` выделен `if (!aggstate->all_cols_needed)`

Поэтому при записи на диск мы оставляем только те атрибуты, которые действительно требуются. А чтобы не сломать существующий код мы просто помечаем лишние атрибуты `NULL`.

> Слайд: `addHyperLogLog`

Во-вторых, не забываем обновить статистику - HyperLogLog, количество кортежей. Но не забываем, что так как все кортежи идут в эту партицию, то какой-то префикс их хэша равен одинаковый, поэтому мы еще раз вычисляем хэш для хэша, чтобы сделать распределение более равномерным.

> Слайд: `LogicalTapeWrite`

И в-третьих, чтобы лишний раз не рассчитывать хэш-значение мы записываем и его вместе с самим кортежом.

> Слайд: `lookup_hash_entries`

Кортеж сбросили на диск и теперь осталось только выставить `NULL` в `pergroup`, чтобы функция перехода не применялась.

> Слайд: `agg_fill_hash_table`

Выходим обратно, вызываем `advance_aggregates` и затем по новой читаем кортеж. Так мы делаем до тех пор, пока не достигнем конца.

По окончании, вызываем `hashagg_finish_initial_spills`, которая сейчас имеет смысл.

> Слайд: `hashagg_spill_finish`

Когда мы сбрасывали на диск, то использовали структуру `HashAggSpill`, но для перезаполнения хэш-таблицы использовать ее не очень удобно. Например, во время работы нам уже не нужно состояние HyperLogLog, а только значение кардинальности. Кроме этого, некоторые партиции могут оказаться пустыми и их бесполезно будет дальше обрабатывать.

Поэтому сейчас мы партиции запечатываем - для каждой создаем создаем свою отдельную структуру `HashAggBatch`. И она уже готова к использованию: статистика подсчитана, LogicalTape готов к чтению. Полученный `HashAggBatch` мы записываем в очередь, из которой будем в последствии читать и заполнять хэш-таблицу.

> Слайд: архитектура disk-spill обработки (`hash-agg-arch`)

Общая картина выглядит следующим образом:

1. На вход нам подается кортеж, для которого мы рассчитываем хэш
2. Если место есть, то записываем в хэш-таблицу, из которой в конечном счете он подастся другому узлу плана.
3. В противном случае, мы идем по правой ветке
4. Для хэш-таблицы создается свой HashAggSpill, в котором хранятся партиции
5. В конце для каждой партиции создается свой HashAggBatch, который кладется в очередь
6. Каждый AggBatch указывает куда-то во временный файл, в котором хранятся его кортежи
7. Затем поочередно мы из очереди получаем батч и заполняем хэш-таблицу повторяя процесс

Можно сразу обнаружить крайний случай, когда каждый кортеж больше ограничения памяти work_mem даже с учетом hash_mem_multiplier. Но это также учтено - использование памяти мы проверяем после заполнения хэш-таблицы, поэтому потихонечку будем совершать прогресс.

Когда мы первый раз заполняли таблицу, то использовали `agg_fill_hash_table`, но для обработки батчей выделена другая функция `agg_refill_hash_table`.

> Слайд: `agg_refill_hash_table`

Код внутри практически идентичен тому, что в `agg_fill_hash_table` за исключениями, что уже оговорили:

1. Чтение кортежей происходит из партиции, созданного тейпа для партиции
   > Слайд: `hashagg_batch_read`

2. При получении элемента хэш-таблицы мы сразу передаем сохраненное хэш-значение
   > Слайд: строка с `LookupTupleHashEntryHash` стрелка от `hashagg_batch_read(&hash)`

> Слайд: `agg_refill_hash_table`, строка с `hashagg_spill_init` выделить `batch->used_bits`, `batch->input_card`.

Также теперь при сбросе нам нужно учитывать новую статистику по этой партции. Эта статистика уже записана в `HashAggBatch`, когда мы запечатывали `HashAggSpill`.

Повторяем это все до тех пор пока не закончится вход. Затем проходимся по хэш-таблице, финализируя агрегаты и снова идем в очередь за батчем. Повторяем так пока очередь не окажется пустой.

И это вся стратегия хэширования.

## Сложные аналитические функции: GROUPING SET, CUBE, ROLLUP

> Слайд: GROUPING SETS, CUBE, ROLLUP и примеры каждого

```sql
EXPLAIN SELECT a, b, c FROM tbl GROUP BY GROUPING SETS(a, (b, d), ROLLUP(a, b, c));
                        QUERY PLAN                        
----------------------------------------------------------
 MixedAggregate
   Hash Key: a, b, c
   Hash Key: a, b
   Hash Key: a
   Hash Key: a
   Hash Key: b, d
   Group Key: ()
   ->  Seq Scan on tbl
```

Для аналитических задач стандарт SQL определяет специальные функции: GROUPING SETS, CUBE и ROLLUP. Идея в том, что мы одним выражением можем определить сразу несколько группировок по разным атрибутам, вместо того, чтобы писать несколько запросов и затем объединять их результаты.

> Слайд: запрос внизу

Самый простой - это GROUPING SETS. Он определяет список выражений, по которым поочередно будет производиться группировка.

Для наглядности в документации PostgreSQL приводится такой пример: находим общую сумму продаж по бренду и размеру (каждого в отдельности) и общие продажи.

В итоге в этом запросе 3 GROUPING SET'а: brand, size и плоская группировка. Если по этому атрибуту нет группировки, то атрибут равен `NULL`.

```sql
SELECT brand, size, sum(sales)
   FROM items_sold
   GROUP BY GROUPING SETS ((brand), (size), ());

 brand | size | sum
-------+------+-----
 Foo   |      |  30
 Bar   |      |  20
       | L    |  15
       | M    |  35
       |      |  50
(5 rows)

          QUERY PLAN          
------------------------------
 MixedAggregate
   Hash Key: brand
   Hash Key: size
   Group Key: ()
   ->  Seq Scan on items_sold
(5 rows)
```

> Слайд: запрос снизу

Этот же запрос мы можем переписать таким образом с помощью UNION ALL:

```sql
-- brand
SELECT brand, NULL, sum(sales)
      FROM items_sold
      GROUP BY brand

UNION ALL

-- size
SELECT NULL, size, sum(sales)
      FROM items_sold
      GROUP BY size

UNION ALL

-- brand AND size
SELECT NULL, NULL, sum(sales)
      FROM items_sold
      GROUP BY brand, size;
```

Остальные 2 функции мы можем определить в терминах GROUPING SETS.

> Слайд: запрос снизу

ROLLUP - группировка по всем возможным префиксам списка выражений. То есть поочередно удаляем по одному атрибуту с конца.

```sql
ROLLUP(a, b, c)

GROUPING SETS(
   (a, b, c),
   (a, b),
   (a),
   ()
)
```

Пустые скобки означают группировку по всем атрибутам. Ее можно использовать даже в обычном запросе, без GROUPING SETS.

> Слайд: запрос снизу

Если применить к той же таблице, то получим следующий результат:

```sql
SELECT brand, size, sum(sales)
   FROM items_sold
   GROUP BY ROLLUP (brand, size);
   
 brand | size | sum 
-------+------+-----
       |      |  50
 Foo   | M    |  20
 Bar   | L    |   5
 Bar   | M    |  15
 Foo   | L    |  10
 Foo   |      |  30
 Bar   |      |  20
(7 rows)

                             QUERY PLAN                             
--------------------------------------------------------------------
 MixedAggregate  (cost=0.00..35.26 rows=401 width=72)
   Hash Key: brand, size
   Hash Key: brand
   Group Key: ()
   ->  Seq Scan on items_sold  (cost=0.00..18.50 rows=850 width=68)
(5 rows)
```

> Слайд: запрос снизу

И последняя функция - CUBE - она генерирует все возможные перестановки.

```sql
CUBE(a, b, c)

GROUPING SETS(
    (a, b, c),
    (a, b   ),
    (a,    c),
    (a      ),
    (   b, c),
    (   b   ),
    (      c),
    (       )
)
```

Опять пример с той же таблицей:

```sql
SELECT brand, size, sum(sales)
   FROM items_sold
   GROUP BY CUBE (brand, size);
   
   
SELECT brand, size, sum(sales)
   FROM items_sold
   GROUP BY GROUPING SETS (
      (brand, size),
      (brand      ),
      (       size),
      (           )
   );

         QUERY PLAN          
-------------------------------
 MixedAggregate
   Hash Key: brand, size
   Hash Key: brand
   Hash Key: size
   Group Key: ()
   ->  Seq Scan on items_sold
(6 rows)

 brand | size | sum 
-------+------+-----
       |      |  50
 Foo   | M    |  20
 Bar   | L    |   5
 Bar   | M    |  15
 Foo   | L    |  10
 Foo   |      |  30
 Bar   |      |  20
       | L    |  15
       | M    |  35
(9 rows)
```

Можем заметить, что все эти функции мы можем переписать с помощью GROUPING SETS. И перед тем как приступить к основному действию, обговорим несколько оставшихся моментов.

> Слайд: запрос снизу

Первое, обычные выражения группировки - это вырожденный GROUPING SETS с 1 выражением. Например, эти 2 запроса идентичны:

```sql
explain select a, b from tbl group by a, b;
                         QUERY PLAN                          
-------------------------------------------------------------
 HashAggregate  (cost=40.60..42.64 rows=204 width=8)
   Group Key: a, b
```

> Слайд: запрос снизу

И третье, GROUPING SETS могут быть вложенными, а вот CUBE и ROLLUP нет. То есть запрос выше будет корректен:


А ниже уже нет, так как ROLLUP не может быть вложенным.


И это не технический недостаток - такое поведение описывает стандарт SQL: GROUPING SETS может содержать любой валидный список группировки (включая себя), а ROLLUP и CUBE только ссылки на элементы SELECT.

Теперь, задача упрощается, так как все представляется в виде GROUPING SETS, то нам остается вначале привести их к этому виду, а затем получить плоский список набора группировок.

### Хэширование GS

> Слайд: запрос с EXPLAIN и от него стрелки к разным хэш-таблицам идут

```sql
EXPLAIN SELECT a, b FROM tbl
   GROUP BY GROUPING SETS((a, b), c, (c, d));

                        QUERY PLAN                        
----------------------------------------------------------
 HashAggregate  (cost=1.06..1.19 rows=10 width=16)
   Hash Key: c, d
   Hash Key: c
   Hash Key: a, b
   ->  Seq Scan on tbl  (cost=0.00..1.04 rows=4 width=16)
(5 rows)
```

Для него все просто - у нас на руках имеется множество отдельных атрибутов группировки. Так как мы умеем работать с 1 группой, то для каждой GROUPING SETS создадим свою хэш-таблицу и будем работать с каждой по отдельности. Единственное, что мы будет шарить - spill режим и файл на диске.

> Слайд: `lookup_hash_entries` с ->num_hashes (+ и другие для наглядности показать)

На этой слайде можете видеть общую идею поддержки множества GROUPING SETS. Все структуры для каждого GROUPING SET мы храним в большом массиве, проиндексированном номером этого GROUPING SET,а в цикле проходим по всем и обрабатываем каждый отдельно.

Для хэширования в переменной `num_hashes` хранится количество этих самых хэш-таблиц для каждого GROUPING SETS, а отдельное состояние - это `AggStatePerHash`. Часто в циклах обхода всех GROUPING SET итерационная переменная называется `setno`.

До этого момента, весь код был показан в предположении, что мы работали только с одной хэш-таблицей, но на самом деле, он состоит из подобных циклов, проходящих по всем GROUPING SET. Даже, если там только 1 хэш-таблица.

И это мы делаем когда настраиваем pergroup массив перед вызовом `advance_aggregates`.

> Слайд: `hashagg_finish_initial_spills`

Когда запечатываем сброшенные партиции в `finish_initial_spills`.

> Слайд: `hash_agg_enter_spill_mode`

А при входе в spill режим `HashAggSpill` создается для каждой хэш-таблицы.

В остальном логика работы не меняется.

### Сортировка GS

> Слайд: запрос с ROLLUP

```sql
SELECT a, b, c FROM tbl GROUP BY ROLLUP(a, b, c);

GROUPING SETS(
   (a, b, c),
   (a, b   ),
   (a      ),
   (       )
)

                           QUERY PLAN                           
----------------------------------------------------------------
 GroupAggregate  (cost=1.08..1.22 rows=10 width=12)
   Group Key: a, b, c
   Group Key: a, b
   Group Key: a
   Group Key: ()
   ->  Sort  (cost=1.08..1.09 rows=4 width=12)
         Sort Key: a, b, c
         ->  Seq Scan on tbl  (cost=0.00..1.04 rows=4 width=12)
(8 rows)
```

У хэширования есть хорошая точка параллелизация - отдельный GROUPING SET. Но в случае с сортировкой такой трюк не прокатит, т.к. разные GROUPING SETS могут иметь разные атрибуты группировки и придется выполнять несколько сортировок.

Но все же мы можем оптимизировать ее. Чтобы понять как, посмотрим на структуру ROLLUP. Он разворачивается в несколько GROUPING SETS, каждый раз удаляя по 1 атрибуту с конца. И первое наблюдение - вся эта последовательность может быть отсортирована 1 раз по наибольшей группе, а все остальные будут также отсортированы, т.к. это префиксы сортированной последовательности. Нам просто нужно научится обрабатывать каждый такой GROUPING SET независимо от других.

> Слайд: основная идея сортировки

Когда мы обрабатывали только 1 группу, то нам нужно было обнаружить первый неравный кортеж. Причем нам было все равно какой именно атрибут поменялся - главное факт изменения.

А чтобы понять идею поддержки множества GROUPING SET'ов, надо сделать такой вывод или даже обобщение - когда меняется атрибут под номером N, то это означает конец GROUPING SET размером N, а также всех больших.

Чтобы понять, что я имею ввиду, посмотрим на слайд. Когда мы рассматривали логику для 1 группы, то натыкаясь на этот кортеж финализировали группу `112`. И начиналась группа `121`. Довольно просто. Но представим, что мы обрабатываем 2 GROUPING SET'а - `ab` и `abc`, т.е. другой размером 2.

> Слайд: вычеркнут/заштрихован 3 атрибут

Если мы как-бы вычернем ненужный ему 3-ий атрибут, то увидим, что логика одиночной обработки для него сохраняется - мы должны финализировать и его группу.

> Слайд: вычеркнут/заштрихован 2 атрибут

Но если мы вычернем еще и 2 атрибут, который не поменялся, то увидим, что его время еще не настало и группа не завершилась.

> Слайд: шаг 1

Попробуем также все вручную обработать.

В самом начале у нас нет ничего, поэтому просто берем первый кортеж и делаем его представителем.

> Слайд: шаг 2

Следующий кортеж равен представителю, поэтому продолжаем.

> Слайд: шаг 3

И вот кортеж `112` отличается. У него неравен атрибут под номером 3, поэтому мы финализируем группу размером 3, в данном случае это весь кортеж.

> Слайд: шаг 4

Следующий идет кортеж, у которого отличается 2-ой атрибут, поэтому мы финализируем сразу 2 группы: `abc` и `ab`.

> Слайд: шаг 5

Следующий равен предыдущему

> Слайд: шаг 6

А последний отличается первым атрибутом, поэтому мы финализируем все группы: 1, 2 и 3.

> Слайд: шаг 7

Ну и по окончании входа мы финализируем все, что осталось. Но так как сейчас у нас несколько GROUPING SET'ов, то финализируем всех их, а не только 1 группу.

При всем этом вы можете заметить, что изменили мы только минимальную часть логики. Представитель хранится только только 1, не нужно делать несколько копий, т.к. каждой группе соответствует свой префикс и состояние каждого GROUPING SET'а свое в одном экземпляре.

> Слайд: запрос снизу + выделить Sort, он один

Все что требуется - найти подобные структуры. Но это задача планировщика и в нее мы залезать не будем, но на слайде можете увидеть, что он с этой задачей справляется.

В этом запросе группы в GROUPING SETS не отображаются напрямую в ROLLUP, но планировщик все-таки оптимальную структуру:

```sql
EXPLAIN SELECT a, b, c FROM tbl GROUP BY GROUPING SETS (
   (c, b, a),
   (b, a),
   (a)
);
                           QUERY PLAN                           
----------------------------------------------------------------
 GroupAggregate  (cost=1.08..1.21 rows=9 width=12)
   Group Key: a, b, c
   Group Key: a, b
   Group Key: a
   ->  Sort  (cost=1.08..1.09 rows=4 width=12)
         Sort Key: a, b, c
         ->  Seq Scan on tbl  (cost=0.00..1.04 rows=4 width=12)
(7 rows)
```

> Слайд: `agg_retrieve_direct` из прошлых слайдов - сохранение grp_firsttuple

За стратегию сортировки отвечает функция `agg_retrieve_direct` и если посмотрим внутрь еще раз, то в том цикле обработки кортежей мы ничего о GROUPING SETS не найдем - ни обнуружение их границ, ни даже сравнение префиксов - мы всегда сравнивали кортежи полностью.

Дело в том, что наша модель выполнения итераторная, т.е. мы должны возвращать по 1 кортежу за раз. Поэтому, чтобы также обарабатывать все в потоке без состояния по обнаружении конца группы мы сохраняем текущий и следующий кортеж.

Прямо сейчас мы финализируем кортеж из самого большого GROUPING SET'а.

> Слайд: `agg_retrieve_direct` где `outertuple =` + выделить место `ExecQualAndReset` + только этот if показать и сразу в конце `finailize_aggregates`

Затем, когда нас вызывают еще раз мы выполняем проверку еще раз, но уже для GROUPING SET меньшего размера. Если и они не равны, то финализируем этот GROUPING SET.

А чтобы отслеживать на каком GROUPING SET мы сейчас используется переменная `projected_set`. Которая сбрасывается при начале новой группы.

Вот таким образом мы обработаем несколько GROUPING SET за 1 проход в потоке.

> Слайд: запрос снизу + выделить все Sort Key, их 2

Но построить такую обработку мы можем не всегда. Например в этом запросе такое провернуть не получится, поэтому для группировки по `c` мы вынуждены выполнить еще 1 сортироку:

```sql
EXPLAIN SELECT a, b, c FROM tbl GROUP BY GROUPING SETS (
   (b, a),
   (a),
   (c)
);
                           QUERY PLAN                           
----------------------------------------------------------------
 GroupAggregate  (cost=1.08..1.26 rows=9 width=12)
   Group Key: a, b
   Group Key: a
   Sort Key: c
     Group Key: c
   ->  Sort  (cost=1.08..1.09 rows=4 width=12)
         Sort Key: a, b
         ->  Seq Scan on tbl  (cost=0.00..1.04 rows=4 width=12)
(8 rows)
```

> Слайд: `agg_retrieve_direct` верхнеуровнево

Когда мы смотрели на код `agg_retrieve_direct`, то нам на вход уже подавались отсортированные кортежи и мы не думали откуда они приходят. Точнее, мы знали - из `fetch_input_tuple`, но внутрь не смотрели - теперь самое время.

> Слайд: `fetch_input_tuple`

Внизу мы видим вызов функции `ExecProcNode`. Это та функция, которая вызывает нижний узел и получает кортеж от него. Когда мы выполняем обычный запрос, без GROUPING SET, то используется именно он.

Но вот, кроме нее стоит обратить внимание на 2 поля: `sort_in` и `sort_out`. Они и нужны для передачи кортежей.
Теперь можем взглянуть на общую картину стратегии сортировки.

> Слайд: схема фаз выполнения с примером запроса и GROUPING SET'ами в каждой фазе (`gs-sort-sort_in-sort_out`)

Все выполнение разбито на отдельные этапы, которые называют фазами. Каждая описывается структурой `AggStatePerPhase`. В каждой фазе находится несколько GROUPING SET'ов, но таких которые мы можем обработать одновременно.

На слайде можете видеть уже более наглядный пример взаимосвязи фаз друг с другом. И связующим звеном в этой истории являются эти 2 очереди: `sort_in` и `sort_out`.

> Слайд: запрос внизу + EXPLAIN

Для махины из схемы наш запрос должен выглядеть примерно так:

```sql
EXPLAIN SELECT a, b, c, d, e, i, j, k FROM tbl1 GROUP BY GROUPING SETS(ROLLUP(a, b, c), ROLLUP(d, e), ROLLUP(i, j, k));
        QUERY PLAN                              
--------------------------
 GroupAggregate
   Group Key: d, e
   Group Key: d
   Group Key: ()
   Group Key: ()
   Group Key: ()
   Sort Key: a, b, c
     Group Key: a, b, c
     Group Key: a, b
     Group Key: a
   Sort Key: i, j, k
     Group Key: i, j, k
     Group Key: i, j
     Group Key: i
   ->  Sort
         Sort Key: d, e
         ->  Seq Scan on tbl1
(17 rows)
```

Для каждой фазы представлен свой Sort Key и внизу написаны их ключи группировки - это и есть отдельные фазы.

> Слайд: вперед, еще раз архитектуру на пред слайде которая

Если не учитывать порядок следования фаз, логика обработки идентичная.

> Слайд: `fetch_input_tuple` выделен sort_in

Через `sort_in` мы получаем кортежи от предыдущего этапа, причем они отсортированы в том, порядке, который нужен нашей фазе.

> Слайд: `fetch_input_tuple` выделен `sort_out`

После же чтения мы этот кортеж должны передать на следующий этап, для этого мы кладем его в `sort_out`. При переходе к следующей фазе выполняем сортировку и просто меняем `sort_in` и `sort_out` местами.

### MixedAggregate

> Слайд: MixedAggregate и план с ним внизу

Теперь мы знаем как можно обработать несколько GROUPING SETS разными стратегиями, а также как переделать множество вложенных и сложных GS/CUBE/ROLLUP в список из обычных GS. В теории мы умеем обрабатывать любой запрос, но подход на выборе исключительной стратегии может дать плохой результат, когда мы выбираем либо хэширование, либо сортировку.

Для решения этой проблемы была придумана стратегия MixedAggregate. Ее идея в том, что мы можем объединить в 1 узле и хэширование, и сортировку, причем выбирать какая из стратегий будет выгоднее для каждой группы в отдельности.

Если мы посмотрим на план этого запроса, то увидим, что она из себя представляет.

```sql
CREATE TABLE tbl(a int, b int);
CREATE INDEX ON tbl(a, b);
INSERT INTO tbl VALUES (1, 1), (1, 2), (1, 3), (1, 4), (1, 5), (1, 6);
SET random_page_cost TO 0;

EXPLAIN SELECT a, b from tbl3 group by cube(a, b);
                                     QUERY PLAN                                     
------------------------------------------------------------------------------------
 MixedAggregate  (cost=0.13..0.41 rows=14 width=8)
   Hash Key: b
   Group Key: a, b
   Group Key: a
   Group Key: ()
   ->  Index Only Scan using tbl3_a_b_idx on tbl3  (cost=0.13..0.22 rows=6 width=8)
(6 rows)
```

Планировщик увидел, что у нас уже есть индекс с нужной сортировкой, но для последнего GROUPING SET он решил использовать хэширование.

> Слайд: фаза хэширования единственная (`gs-hash-phase`)

В архитектуре с фазами хэширование может описаться единственным узлом, так как он может обработать вообще все GROUPING SET за раз. Но в MixedAggregate используется как сортировка так и хэширования. Остается вопрос - как это все скомбинировать? Фаза хэширования одна, поэтому надо решить куда вставить ее.

В начало не вариант, так как на вход могут подаваться уже отсортированные данные, поэтому нам придется выполнять сортировку еще раз. Куда-то в между фазами сортировки тоже не лучшая затея, так как все передается через сортировку, а хэшированию она не нужна.

Остается последний вариант - вставить в конец. Но опять вспоминаем - все передается через сортировку, но нам она не нужна. Для решения этой задачи мы разделяем логику работу хэширования на 2 части: изначальное заполнение таблицы и ее обработку.

> Слайд: `agg_retrieve_direct` с `== AGG_MIXED && current_phase == 1`

Таблицу мы заполняем в самой первой фазе, когда выполняется фаза сортировки. Внутри `agg_retrieve_direct` мы одновременно обрабатываем и сортировку и хэширование. Когда мы прочитали кортеж и видим, что находимся в первой фазе смешанного агрегирования, дополнительно обрабатываем хэширование.

И под конец первой фазы у нас имеется заполненная хэш-таблица и, возможно, сброшенные партиции.

> Слайд: `ExecAgg` с разделением AGG_HASHED и AGG_MIXED

Когда мы доходим до последней фазы, то вместо `AGG_HASHED` там стоит `AGG_MIXED`, который пропускает этап заполнения таблицы и сразу начинает читать из нее.

> Слайд: `ExecAgg` выделен phase->

Как вы можете догадаться, нам нужно различать разные фазы друг от друга, но при этом понимать как у нас глобальная стратегия. Поэтому у каждой фазы имеется своя отдельная стратегия, которой он придерживается.

> Слайд: схема MixedAggregate, этап 1

Таким образом мы пришли к полной картине обработки MixedAggregate.

В самом начале кортежи мы получаем из какого-то подузла. Этот вход мы передаем фазе AGG_SORTED и одновременно с этим заполняем хэш-таблицы.

> Слайд: схема MixedAggregate, этап 2

После этого обрабатываем каждый следующий AGG_SORTED как раньше, передавая кортежи через sort_in/sort_out.

> Слайд: схема MixedAggregate, этап 3

В самом конце мы приходим к AGG_MIXED. Хэш-таблица уже заполнена и нам остается только обработать ее.

На этом мы и закончим разбор архитектуры группировки. Теперь можем составить общую картину работы группировки.

> Слайд: несколько GROUP BY, стрелка вправо - в какие GS разворачиваются

```sql
SELECT avg(a) FROM tbl;
=> ()

SELECT a, b, c FROM tbl GROUP BY a, b, c;
=> (a, b, c)

SELECT a, b, c FROM tbl GROUP BY GROUPING SETS(a, b, c);
=> (a) 
   (b)
   (c)

SELECT a, b, c FROM tbl GROUP BY a, GROUPING SETS(b, ROLLUP(c, d));
=> (a, b)
   (a, c, d)
   (a, c)
   (a)

```

GROUP BY на входе разбиваем на множество из отдельных GROUPING SET.

> Слайд: тот же пример, но напротив AGG_SORTED или AGG_HASHED

Затем планировщик определяет какую стратегию лучше все использовать для каждой группы и определяется глобальная стратегия.

> Слайд: кликаю и остается только хэширование

Eсли все эффективнее выполнить хэшированием...

> Слайд: explain с запросом и стрелки к каждой хэш-таблицей

...то для каждого GROUPING SET создаем свою хэш-таблицу и работаем с каждой по отдельности.

> Слайд: архитектура хэширования

Если память переполнена то инициируем spill режим и записываем кортежи, групп для которых нет, на диск в свои партиции.

Каждая партиция расчитывается таким образом, чтобы из нее можно было построить отдельную хэш-таблицу, которая бы полностью умещалась в памяти.

Записываем кортежи на диск, а после вставляем эту партицию в очередь на обработку. Затем заново обрабатываем партицию и если необходимо еще раз сбрасываем кортежи и кладем новую партицию в очередь. И так до тех пор пока не будет обработан каждый кортеж.

> Слайд: открыт столбец AGG_SORT

Если нужно сортировка, то ...

> Слайд: схема sorted agg (была ранее)

... все GROUPING SET разбиваем на отдельные фазы, которые можем обработать за 1 проход/сортировку, то есть имеют общий префикс.

Кортежи сначала берем из подузла, а затем передаем другой фазе с помощью временного файла, на котором будет происходить сортировка.

> Слайд: простая логика группировки (без GS)

Внутри каждой фазы мы запоминаем текущий кортеж и читаем до тех пор пока не найдем неравный. После этого финализируем группу и начинаем новую.

> Слайд: логика с GS

Если у нас есть несколько GROUPING SETS, то доходим до первого неравного кортежа и начинаем сравнивать все префиксы с кортежом предыдущей группы. Сравниваем с конца и для каждого неравного префикса финализируем агрегат и начинаем новую группу.

Каждую такую фазу мы обрабатываем в потоке за один проход.

> Слайд: схема с MixedAggregate (ранее была)

В начале располагаем стретегии сортировки, но на первой фазе мы дополнительно заполняем хэш-таблицу.

В конце, когда закончили с сортировкой, начинаем фазу хэширования, но так как таблица уже заполнена, то сразу переходим к логике ее обработки.
